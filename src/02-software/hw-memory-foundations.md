# 硬件侧内存语义基础

!!! abstract "本节定位"
    本节从硬件视角阐述超节点 Scale-Up 域内的**内存事务语义、保序机制与地址翻译**问题。它不是某一厂商的实现分析（NVIDIA 实现详见下一节"统一寻址与内存语义"），而是从协议层面建立通用的理解框架——无论底层采用 NVLink、UALink、ETH-X 还是其他 Scale-Up 互联方案，这些硬件语义问题都是必须回答的。

## 为什么需要讨论硬件侧内存语义

在单 GPU 内部，SM 核心发出的 Load/Store 指令经过片上网络（NoC）直接到达 HBM 控制器，事务的保序、原子性和一致性由芯片内部的硬件机制保障，对软件完全透明。

当 Scale-Up 互联将多个 GPU 的显存纳入同一个高带宽域时，原本"芯片内"的问题变成了"芯片间"甚至"机柜间"的问题。访存事务从一颗 GPU 出发，经过 IO 模块、交换网络、到达远端 GPU，再原路返回响应——这条路径上的每一跳都可能引入乱序、丢包和延迟不确定性。如果硬件层面不能正确处理这些问题，上层软件无论怎么优化都无法保证程序的正确性。

从帕累托前沿外推的角度看，Scale-Up 域内的内存语义能力决定了"硬件打开的可能性空间有多大"：支持的语义越完整、保序机制越高效，软件能兑现的 Goodput 上限就越高。反之，如果硬件只支持粗粒度的 DMA 拷贝语义而不支持细粒度的内存语义访问，整个系统的编程模型和通信效率都会受到根本性限制。

## 两种基本访存语义

Scale-Up 域内的 GPU 间数据交换有两种基本模式，它们对硬件的要求截然不同：

### Direct Copy（先拷贝，后使用）

GPU 内的 DMA 拷贝引擎（Copy Engine / TMA）将本地 HBM 中的数据批量搬运到远端 GPU 的 HBM 中，计算引擎随后从远端 HBM 加载数据使用。

| 特性 | 说明 |
|:-----|:-----|
| **数据粒度** | 大块连续数据（KB 至 MB 级） |
| **启动延迟** | 较高（需要设置 DMA 描述符） |
| **带宽利用率** | 高（大包传输，协议开销低） |
| **发起方** | 专用硬件拷贝引擎，不占用 SM 算力 |
| **典型场景** | 粗粒度的计算通信重叠（计算 Kernel 与通信 Kernel 分离） |

Direct Copy 语义对应 RDMA 技术中的 `PUT/GET` 操作，在 ETH-X 协议中由 IO 模块的 AXI 事务引擎或专用 Scale-Up 拷贝引擎（SU Engine）承载。

### Direct Access（边读写，边使用）

GPU 的 SM 核心通过 Load/Store 指令直接访问远端 GPU 的 HBM，数据从远端直接加载到 SM 的寄存器或共享内存中，无需经过本地 HBM 中转。

| 特性 | 说明 |
|:-----|:-----|
| **数据粒度** | 细粒度、非连续数据（1B 至数十 KB） |
| **启动延迟** | 极低（由 SM 核心直接发起） |
| **带宽利用率** | 取决于访问模式（随机访问时可能较低） |
| **发起方** | SM 核心，占用算力但免去中转开销 |
| **典型场景** | 计算通信融合 Kernel（如 FLUX 框架）、细粒度数据依赖 |

Direct Access 语义要求 Scale-Up 互联提供**内存语义级**的端到端传输能力：IO 模块必须能将 SM 发出的 AXI 总线事务封装成网络数据包，在远端正确执行后返回响应。

### 两种语义的互补关系

两种语义并非替代关系，而是针对不同通信模式的互补：

```
计算 Kernel 输出 → 通信 Kernel 搬运（Direct Copy）
          适合粗粒度、结构化、可预测的通信

计算/通信融合 Kernel → SM 直接发起远端访问（Direct Access）
          适合细粒度、非连续、延迟敏感的通信
```

对 Scale-Up 互联硬件的要求是：**必须同时支持两种语义**。只支持 Direct Copy 的互联方案（如传统 RDMA/RoCEv2）可以处理 AllReduce 等批量集合通信，但无法支持融合 Kernel 等新一代编程范式对细粒度远端访存的需求。

## 内存事务保序问题

当访存事务从 GPU 经过 Scale-Up 网络到达远端 GPU 时，事务的执行顺序可能被网络传输和接收端处理改变。如果保序机制不正确，程序将产生数据竞争和不一致。

### 单指令流内的程序顺序

同一指令流（例如 NVIDIA GPU 上的一个 Warp）内部发出的内存访问，硬件必须保证**存在数据依赖的指令之间保序**——先发出的对地址 A 的写操作，必须在后续对地址 A 的读操作之前完成。不存在数据依赖的指令则允许乱序执行以提升吞吐。

**Scale-Up 网络对程序顺序的挑战**：

当事务经过多条并行路径传输时（多平面 Multi-Plane 或负载均衡），不同路径的传输延迟存在差异，可能导致后发起的事务先到达。如果两个事务访问相同地址且存在数据依赖，这种乱序将破坏程序正确性。

```
GPU_A 发出事务序列：
  T1: STORE addr=X, data=100     (先发)
  T2: STORE addr=X, data=200     (后发)

经过负载均衡分到不同路径：
  T1 → 路径 A (拥塞，延迟较高)
  T2 → 路径 B (空闲，延迟较低)

远端 GPU_B 观察到的执行顺序：
  T2 先到达 → addr=X 被写为 200
  T1 后到达 → addr=X 被写为 100  ← 错误！最终值应为 200
```

**解决方案**：相同指令流内对相同地址的事务必须在网络中以相同路径传输，获得相同的 Flow Entropy 标识，使交换机在负载均衡时将它们路由到同一条路径。

### AXI 总线的读写乱序

Scale-Up IO 模块通常直接接入 GPU 的 AXI 总线。AXI 协议的读通道和写通道是独立的，在传输和处理过程中不存在跨通道的顺序约束。这导致两类乱序：

- **写后读（RAW）乱序**：先发出的写事务和后发出的读事务访问相同地址，读可能在写完成前返回旧数据
- **读后写（WAR）乱序**：先发出的读事务和后发出的写事务访问相同地址，写可能在读完成前覆盖数据

IO 模块的 AXI 适配器需要实现**同地址保序引擎**，检测跨通道的地址冲突并强制排序。

### 释放一致性内存模型（Release Consistency）

GPU 不同线程之间的内存访问通常遵循**释放一致性（Release Consistency）**模型。这是一种较弱的内存模型，允许在满足特定同步操作（Acquire/Release）的前提下对不同地址的读写操作进行乱序：

| 语义 | 适用操作 | 保证 |
|:-----|:---------|:-----|
| **Release** | store, fence, atomic | 确保该操作之前的所有内存写入都在该操作完成前完成 |
| **Acquire** | load, fence, atomic | 确保该操作之后的所有内存读取都在该操作完成后执行 |
| **Relaxed** | load, store, atomic | 无跨线程保序约束 |

当两个线程分布在 Scale-Up 域内的不同 GPU 上时，它们通过共享的内存地址空间和 Acquire-Release 同步原语进行通信。**Release 操作之前的所有写入，在另一个线程执行对应的 Acquire 操作后必须变得可见。**

### Scale-Up 域对 Release 语义的三重挑战

**挑战一：网络传输破坏保序**

典型的生产者-消费者场景：GPU_0 写入数据后通过 Release 语义写入 Flag 通知 GPU_1。如果数据写入和 Flag 写入经过不同网络路径，Flag 可能先于数据到达，GPU_1 读到 Flag 后访问到的是未完成的数据。

```
GPU_0 (生产者)：
  STORE data[0..N]       ← Relaxed 写入
  STORE flag = 1         ← Release 写入

GPU_1 (消费者)：
  while (LOAD flag != 1) ← Acquire 读取
  USE data[0..N]         ← 期望看到完整数据
```

如果 Scale-Up 网络允许 Flag 的写入先于 data 的写入到达 GPU_1，消费者将读到不完整的数据。

**挑战二：接收端处理破坏保序**

即使网络传输保序，接收端 GPU 内部也可能引入乱序。不同的写事务可能命中不同的 HBM 内存控制器，这些控制器的执行延迟存在差异，导致先到达的事务后完成。

**挑战三：多 IOD 负载均衡破坏保序**

当一个 GPU 连接多个 IO 模块（IOD）时，事务在多个 IOD 之间进行负载均衡。不同 IOD 及其对应网络平面的处理延迟不同，Release 语义事务可能在某个 IOD 平面上先于 Relaxed 语义事务完成。

### P2P 一致性与全局一致性

满足一对 GPU 之间的 Release 语义（P2P 一致性）是基本要求。但当通信涉及三个或更多 GPU 时，仅有 P2P 一致性可能不够：

```
GPU_0 写入 GPU_1 和 GPU_2 的内存后，通过 Release 写入 GPU_3 的 Flag。
即使 GPU_0-GPU_1 和 GPU_0-GPU_2 各自满足 P2P 一致性，
由于不同 GPU 对的事务完成时间不同，
GPU_3 读到 Flag 时可能 GPU_2 上的数据尚未就绪。
```

全局一致性要求：Release 操作前针对**所有目标 GPU** 的写操作都必须完成。实现方式是在源端 GPU 执行**系统级内存屏障（System Fence）**——当指令流触发 Fence 时，GPU 核心暂停当前指令流的执行，直至所有先前写操作的真实响应返回（包括事务代理触发的脏数据写回）。

全局一致性的执行开销显著高于 P2P 一致性。因此，Scale-Up 系统可以提供**基于区域的一致性域（Region-based）**编程接口，由用户根据通信模式选择适当的一致性级别以优化同步开销。

## 事务代理与延迟优化

GPU 核心在发起访存事务时维护一组事务队列（Outstanding Buffer），事务元数据在请求发出到响应返回之前一直占用队列槽位。当队列满时，核心被阻塞，无法继续发出后续请求。

在 Scale-Up 域内，事务的端到端响应延迟是本地访存的十倍以上。如果所有域内事务都必须等待跨 Scale-Up 域的完整往返后才释放队列槽位，核心的并发吞吐将大幅下降。

**事务代理（Transaction Proxy）** 解决这一问题：对于 Push 类型事务（Store/Write），在 GPU 本地 Scale-Up 出口侧设置代理缓冲区。代理接收事务后立即向核心返回快速响应（释放队列槽位），事务在 Scale-Up 域上的后续传输由代理负责。这将 Push 事务的响应延迟从"跨域往返"缩短到"本地代理往返"。

Pull 类型事务（Load/Read）无法使用代理——必须等待远端实际返回数据后才能响应。因此，**Pull 事务的延迟直接决定了 Direct Access 模式下的性能上限**。

```
无代理：
  SM → [IO模块] → Scale-Up网络 → 远端GPU → 响应返回
  ◄──────────── 完整往返延迟 ────────────►  队列占用全程

有代理（Push 事务）：
  SM → [IO模块/代理] → 快速响应
  ◄── 本地延迟 ──►  队列快速释放
        代理 → Scale-Up网络 → 远端GPU  (后台完成)
```

事务代理需要正确处理对已缓存地址的 Pull 事务以及内存屏障指令，以保证数据正确性和一致性。

## 专用 Scale-Up 拷贝引擎

对于 Direct Copy 语义，使用 GPU 原生的本地拷贝引擎通过 AXI 总线发起远端数据搬运存在多重低效：

1. **总线流量翻倍**：数据先从本地 HBM 读出到拷贝引擎，再通过 AXI 总线发送到 IO 模块——同一数据穿越总线两次
2. **切分-聚合开销**：本地拷贝引擎将大消息切分为小粒度 AXI 子事务，IO 模块收到离散子事务后需要二次聚合成网络大包
3. **不必要的保序约束**：本地拷贝引擎的 AXI 子事务受 GPU 内存模型的全局保序约束，但 Direct Copy 语义本身具备原子性，内部子传输可以使用独立的、更宽松的顺序模型
4. **内存屏障干扰**：系统级内存屏障会阻塞总线上所有子事务（包括拷贝引擎的），导致不必要的停顿

在 IO 模块内实现**专用 Scale-Up 拷贝引擎（SU Engine）** 可以避免上述问题：数据直接从本地 HBM 读取并封装成大包发送，无需经过 AXI 总线中转；子传输顺序由拷贝引擎独立管理；Copy 语义与 Access 语义之间通过专用内部屏障进行隔离，互不干扰。

## MMU 与地址翻译

Scale-Up 域内的统一访存需要**将多个 GPU 的物理内存映射到全局统一的虚拟地址空间**。这依赖 GPU 内部的 MMU（Memory Management Unit）和 Scale-Up IO 模块的地址翻译能力。

### GPU 侧 MMU

GPU 的 MMU（如 NVIDIA 的 GMMU）通过多级页表将 GPU 虚拟地址（GVA）翻译为物理地址。页表项（PTE）中的关键字段决定了访问路由：

| PTE 字段 | 作用 |
|:---------|:-----|
| **Aperture** | 指示目标地址空间类型：本地显存（VIDEO）、远程 GPU（PEER）、系统内存（SYS） |
| **Peer Index** | 当 Aperture=PEER 时，标识目标 GPU 的出口端口编号 |
| **Physical Address** | 目标物理地址；对于 Fabric 访问，此字段编码全局 Fabric 地址 |

对于 Scale-Up 域内的远端访问，PTE 配置为 `Aperture=PEER`，物理地址字段填入目标 GPU 的**全局 Fabric 地址**（Fabric Base Address + 本地偏移）。MMU 硬件不区分本地 P2P 和跨节点 Fabric 访问——路由决策完全由物理地址中编码的 Fabric 信息和交换芯片的路由表完成。

### IO 模块的地址映射

Scale-Up IO 模块（IOD）需要将 GPU AXI 总线上的事务目标地址翻译为网络地址。典型实现采用**寄存器映射表**：

- AXI 总线事务携带 User 信号字段，其中编码了目标 GPU ID
- IO 模块通过查找 DA Mapping 寄存器，将 GPU ID 映射为目标 GPU 的网络地址（MAC 地址或 Fabric 地址）
- 映射表在系统启动阶段由上层软件通过 APB 接口配置

### 统一虚拟地址（UVA）

统一虚拟地址技术允许 Scale-Up 域内的所有 GPU 共享同一个虚拟地址空间，使多个 GPU 可以直接互相访问对方的内存。UVA 通过硬件级页表同步实现内存地址的统一管理。

UVA 的实现存在两种模式：

| 模式 | 机制 | 适用场景 |
|:-----|:-----|:---------|
| **硬件统一编址** | GPU 间通过 Scale-Up 互联直接互访，MMU 硬件自动路由 | Scale-Up 域内，要求高带宽低延迟 |
| **PGAS（分区全局地址空间）** | 逻辑统一但物理分区，借助 OpenSHMEM/NVSHMEM 等实现 | 跨域或跨 OS 实例场景 |

当超节点的多个 GPU 分布在不同物理服务器上、运行在不同 OS 实例中时，需要在多个 OS 之上建立统一的虚拟地址空间。这通常需要额外的 Fabric MMU API 抽象层来简化编程和系统管理。

## 对参考设计的含义

!!! info "对参考设计的含义"

    硬件侧内存语义的支持程度，直接决定了各参考设计在帕累托七维度中"单跳/访存延迟"和"软件复杂度"两个维度上的位置：

    - **标准以太网方案（ESUN/ETH-X）**：通过 PAXI 事务层在以太帧上承载 AXI 事务，实现 Direct Access 语义。但以太帧封装引入额外开销（传输效率约 56%→74%），需要 LLR + CBFC 保障可靠性，内存保序由端侧 IOD 的 AXI 适配器和事务代理负责。协议复杂度高但生态兼容性好。
    - **标准总线方案（UALink）**：原生 Flit 设计，固定 640B 微包免去帧头解析，内存语义从协议底层原生支持。保序和一致性机制与 NVLink 方法论接近，链路利用率可达 93%。但交换芯片尚未量产。
    - **NVLink/NVSwitch（帕累托基准）**：完整的硬件级内存语义、<100ns 单跳延迟、Fabric Manager 协调的全局地址空间。保序和一致性由 NVSwitch 硬件路由表和 GPU GMMU 协同保障，是当前工程验证最充分的方案。

    **核心 trade-off**：以太网路线为了兼容开放生态，在协议层重新实现了总线内原生具备的内存语义——代价是更高的协议复杂度和延迟开销。总线路线原生支持内存语义，但需要专用交换芯片，生态成熟度较低。这一 trade-off 是第四章帕累托七维度评分中"延迟"与"生态成熟度"之间此消彼长关系的硬件根源。
