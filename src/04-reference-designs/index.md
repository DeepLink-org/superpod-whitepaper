# 参考设计

本章的任务，是把前文关于硬件边界、软件兑现与建模仿真的讨论，收束为"如果今天要落地，一个超节点方案应该如何选、如何组织、适合什么场景"的参考设计问题。前几章回答的是"有哪些约束"和"如何评估取舍"，而本章要回答的是"在这些约束和评估方法之下，当前阶段有哪些可行方案空间"。

需要强调的是，参考设计不是单点最优答案，而是约束条件下的方案空间。同一技术路线很难同时覆盖所有场景——规模、时延、带宽、功耗、成熟度与软件复杂度会共同决定方案边界。每个方案都必须同时回答两个问题：今天能否落地，以及未来是否有清晰的演进路径。

## 方案分类

本章将参考设计分成两类：

- **标准构型**：以当前更成熟的工程路径为核心，强调可落地性、生态兼容与可运维性。
- **探索构型**：面向更大规模、更低时延或更高带宽目标，在拓扑与光交换等方向上主动引入更强的系统重构能力。

## 方案选择维度

在本章语境下，参考设计主要沿以下维度比较：

| 维度 | 关注问题 |
|:---|:---|
| 目标规模 | 方案主要面向机内、机柜级还是更大规模 HBD |
| 域内带宽特征 | 更适合全对等、局部高带宽，还是动态重构 |
| 网络直径与时延 | 更强调低跳数、低尾延迟，还是平衡扩展性 |
| 软件与控制面要求 | 对运行时、拓扑管理器、流控与容错机制的要求有多高 |
| 功耗与 TCO | 光电转换、交换层级、布线与运维复杂度是否可控 |
| 成熟度与可运维性 | 当前生态、供应链与维护方式是否足够成熟 |
| 适用负载 | 更适合稠密训练、MoE、长上下文推理还是混合负载 |

## 方案总览

### 标准构型

- [标准以太网方案](standard-ether.md)：以开放以太网生态为基础，面向更高兼容性和更低迁移成本的全对等互联路径。
- [标准总线方案](standard-bus.md)：以总线/内存语义为核心，适用于机柜内更强耦合的高带宽域构建。

### 探索构型

- [Dragonfly + OCS](dragonfly-ocs.md)：以 Dragonfly 分组拓扑为骨架，引入 OCS 作为组间弹性重构能力，适合面向更大规模的组间优化。
- [Torus + OCS](torus-ocs.md)：以 3D Torus 逻辑拓扑为核心，利用 OCS 动态成环与绕障，适合邻近通信友好且强调拓扑弹性的场景。
- [大环路 + 分布式 dOCS](ring-docs.md)：将分布式光交换能力进一步下沉到互连模块，面向更激进的拓扑重构探索。

这些方案不是终点——它们的成立条件仍会被光互联、封装、Chiplet、HBM/3D DRAM 与模型形态演进持续改写。因此下一章将转向未来演进，讨论哪些技术变量最可能重新塑造本章的方案边界。
