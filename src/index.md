# 面向下一代模型的超节点技术白皮书

大模型系统正在进入一个新的约束阶段。过去几年，训练与推理需求沿着近似指数轨道持续抬升，而单芯片算力、显存容量、互联距离、供电散热与软件复杂度却无法同步线性扩展。问题已经不再是“有没有更强的单卡”，而是“如何把足够多的计算、内存与互联资源组织成一个稳定兑现 Goodput 的系统”。

本白皮书讨论的正是这个问题。我们将“超节点”视为一种系统级基础设施形态：在受控物理范围内，把高带宽互联、显存池化、集合通信与运行时控制组织成一个更接近“放大版单机”的高带宽域（HBD），以承接强耦合训练、长上下文推理、MoE 和多模态等负载。

## 这本白皮书回答什么

本书试图回答三个互相关联的问题：

- **为什么超节点正在成为主流形态**：当模型规模和序列长度继续增长，传统“单机 + Scale-Out”的形态为什么越来越难以维持效率。
- **超节点应该如何分析与设计**：在带宽、时延、成本、功耗、软件复杂度与可运维性之间，如何做出工程可行的权衡。
- **未来 2-5 年哪些变量会改写当前方案**：光互联、先进封装、Chiplet、HBM/3D DRAM 与模型形态演进，将如何改变今天的系统设计边界。

## 核心判断

本白皮书采用一个统一判断来组织全文：

> 超节点不是某一种固定产品形态，而是一个持续演进的系统级帕累托优化问题。

它有两层含义：

- **指数增长轨道**：AI 基础设施的性能边界仍在快速外移，任何方案都必须回答“能否跟上下一代平台与软件栈的节奏”。
- **帕累托前沿选择**：在任意时点，系统设计都需要在带宽、成本、规模、功耗、软件复杂度和运维约束之间寻找最优权衡点，而不存在脱离约束的唯一最优解。

因此，评估一个超节点方案，不能只看峰值带宽或理论算力，而要看它能否在真实负载和真实运行环境下，持续把硬件能力兑现成稳定 Goodput。

## 术语与边界

为了避免将“超节点”泛化为“多机集群”的同义词，本文采用以下工作性边界：

- **超节点 / HBD（High Bandwidth Domain）**：在受控物理范围内，通过高带宽互联组织起来的强耦合计算域，对上层更像“放大版单机”。
- **Scale-Up**：在域内扩展带宽、显存和协同效率，解决强耦合通信问题。
- **Scale-Out**：在域间进行更大规模扩展，承担弱耦合任务分发、数据分片与跨域调度。
- **Goodput**：最终被系统稳定兑现出来的有效吞吐，而不是硬件规格上的理论峰值。

本书按“问题定义 -> 约束分析 -> 软件兑现 -> 方法验证 -> 方案设计 -> 未来演进”的顺序展开：

- [架构分析](01-architecture/index.md)：定义超节点是什么、为什么需要它，以及当前硬件/协议/拓扑/系统实现的边界。
- [软件系统](02-software/index.md)：说明统一寻址、通信运行时、RAS 和训推工程如何把硬件能力兑现为 Goodput。
- [建模仿真](03-simulation/index.md)：把争论转化为可验证假设，建立架构选型与优化决策的方法论底座。
- [参考设计](04-reference-designs/index.md)：给出当前阶段可落地的标准构型与探索构型，并说明适用边界。
- [未来演进](05-future/index.md)：讨论光互联、封装、Chiplet、HBM/3D DRAM 和模型形态演进如何改写当前设计。
- [总结](06-conclusion.md)：回收全书判断，形成阶段性结论与路线建议。
