    通信库的架构与设计 ¶

    产出：在分布式共享内存语义下补充通信库，为集合通信补充新型通信语义。

    集合通信语义在超节点上的收益 ¶

    • AllReduce / Broadcast / ReduceScatter / AllGather 在 HBD 域可近似线速，瓶颈在跨 HBD 收敛比与拥塞。
    • 分层通信：HBD 内高频小粒度，HBD 间聚合 / 稀疏化，提升全局吞吐。
    • 对比 BSP 模式：传统 SPMD + 屏障在非均匀负载（MoE、长序列）下放大 straggler 问题，超节点需要更灵活的通信语义与分层调度。

    All-to-All 性能分析 ¶

    • 小包抖动与大包拥塞是 A2A 的主要瓶颈；路由感知 / 负载感知的调度可减少热点。
    • 稀疏 A2A（MoE/ 分段推理）需结合拓扑感知与动态路由。

    PGAS 内存模型与单边通信 ¶

    • Put/Get/ 原子等单边操作将“通信”退化为“内存访问”，天然适应非均匀负载 [@nvshmem_magnum_io; @openshmem15_spec; @pathways_dataflow]。
    • GPU 运行时协同：Pinned 内存、内存注册、RDMA doorbell 映射、完成通知队列等。
    • 混合编程故事线：在 MoE 场景，热专家与冷专家负载差异大——集合通信易被慢专家拖累；使用单边通信将结果直接写入对端内存，再由对端按需消费，降低全局同步成本。

    SPMD/BSP vs PGAS/ 单边通信（重述）¶

    特性 SPMD + 集合通信 (BSP) PGAS + 单边通信
    通信模式 双边：Send/Recv 配对，强同步 单边：Put/Get/ 原子，异步解耦
    性能风险 Straggler 放大，屏障拖累长尾 异步 / 稀疏友好，降低全局同步成本
    适用场景 稠密训练、负载均匀 MoE、长序列推理、异步数据流
    实现要点 集合通信库（NCCL/HCCL） NVSHMEM/PGAS 语义、完成通知、内存注册
